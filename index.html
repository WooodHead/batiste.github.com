Like.js - Introduction

Like.js is micro framework used to define behaviors. A behavior declare how one or many HTML elements will react to some events.

Like.js is a simple standalone library. It is small and easy to understand. I created it for fun and also out of frustration dealing with big and messy JavaScript applications. It's experimental and I never used it for anything real yet but it reflects they way I go about developing a web site with jQuery right now.

A very simple example

Like.js expose an unique object in the global scope: like.
The most important method is like.a. It is used to define a behavior associated with a CSS class. eg:

like.a("popup", "click", function(dom, event){...});

As soon as the popup behavior is declared, all elements in the page with the CSS class like-popup will respond to the click event. The callback function is called with the DOM element as a first parameter and the event as the second.

What's good about it?

Behaviors forces you to define how a DOM element will behave without having the real object on hand. This and the way the code is architectured brings some benefits to some more naive jQuery usage:

  • Separation of concern: context specific behaviors are not possible. Behavior definition are declared wihout being tied to any element in the page. In fact they are only element they need is the body of the page.
  • Inversion of responsability: It's easier to customize a behavior using the data attribute of the HTML element than creating new behaviors for every use case. This helps reusing behaviors and put the settings where it matters: close to the HTML elements that uses them.
  • Easy discovery: When reading the HTML document classes like like-popup will immediatly reveal a behavior and gives the reader an idea of what the behavior is doing in this context.
  • Simplicity: Removing or adding a behavior is as simple as adding/removing the like-popup class. Both can be done easily in the HTML document or directly the browser inspector for debugging.
  • Automatism: Events are automatically handled by a central event delegation dispatcher. You never bind events to any elements.
  • Centralisation and inspection: Defined behaviors are available in a central registry. They can be inspected, removed, added or disabled at will.
  • Dynamic creation of elements is easy: Because there is no events binded on any elements (to the exception of the body where all events are delegated), inserting new elements in the HTML document effortless.

Why using classes and not data attributes?

I wanted to make a standalone framework classes seemed a easier fit as the is native support to fetch classes (read faster) in most browsers.

Classes can also be used for styling, if it make sense in the context of the behavior. Eg. if you have a button behavior, you probably want the button to look the same all accross you application and therefore you could use the like-button class for style in this case.
